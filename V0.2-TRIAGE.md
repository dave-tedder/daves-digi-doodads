# v0.2 Triage: rules → skills vs reference

Decisions for the v0.2-skills-refactor session. Output of triage done in the v0.1.0 ship session.

## Summary

- **22 rules → 24 skills** (20 direct conversions + 2 rules that split into 2 skills each)
- **2 rules stay as reference markdown** in `plugins/goof-proofs/rules/`
- **Total skill count in v0.2:** 29 (24 new + 5 existing)

## Decisions

### Rules that become skills

Each entry: current filename → proposed skill name, then the draft trigger description.

1. **ai-sdk-openai-provider-defaults.md** → `ai-sdk-provider-defaults`
   > Use when writing or editing TypeScript/JavaScript code that imports from `@ai-sdk/openai` or instantiates a language model against an OpenAI-compatible backend (OpenRouter, Groq, Together, DeepInfra, Fireworks). Guards against the silent `/v1/responses` vs `/v1/chat/completions` routing bug.

2. **css-hidden-attribute.md** → `css-hidden-attribute`
   > Use when authoring CSS rules that set `display` on a class, or when debugging an element that has the `hidden` attribute set but still renders visible.

3. **git-commit-splitting.md** → `git-commit-splitting`
   > Use when a single modified file contains hunks that belong to different logical commits (different sessions, different features, different bug fixes) and need to be split into separate commits. Especially relevant when `git add -p` isn't usable (no interactive stdin).

4. **icloud-code-in-markdown.md** → `icloud-code-in-markdown`
   > Use when creating or editing markdown documentation stored on iCloud Drive, when the doc contains substantial fenced code blocks.

5. **icloud-git-fragility.md** → `icloud-git-fragility`
   > Use when working in a git repository located inside iCloud Drive, particularly after an interrupted git operation, when running recursive greps across a multi-worktree directory, or when `.git/objects` corruption symptoms appear.

6. **ios-print-gotchas.md** → `ios-print-gotchas`
   > Use when implementing or debugging print functionality for a web application targeting iOS Safari, especially when printing content that lives inside a modal or fixed-position container.

7. **ios-safari-inputs.md** → `ios-safari-inputs`
   > Use when building or debugging text input fields in a web app for mobile iOS (chat inputs, forms, message composers), particularly if users report page zoom on focus, keyboard layout issues, or confusing form-assistant UI.

8. **llm-decision-scoping.md** → `llm-decision-scoping`
   > Use when designing or debugging any LLM call that asks a model to pick matches from a list (action item resolution, classification, candidate filtering, tagging, matching).

9. **llm-metadata-extraction.md** → `llm-metadata-extraction`
   > Use when writing or debugging an LLM prompt that extracts structured metadata (action items, tags, dates, project fields, categories) from narrative text, or when co-generating creative content alongside a derived structured list.

10. **multi-session-workflow.md** → `multi-session-workflow`
    > Use when evaluating whether a project needs to be split across multiple chat sessions, when starting or closing a tracked project session, or when crafting handoff prompts between sessions.

11. **nextjs-middleware-auth.md** → `nextjs-middleware-auth`
    > Use when implementing or debugging authentication in a Next.js App Router application, especially middleware-based password or token checks, or when a protected route is serving content to anonymous traffic.

12. **node-dev-servers.md** → `node-dev-servers`
    > Use when starting or debugging Node.js development servers (Express + Vite, similar concurrent-process setups) from inside Claude Code, or when sharing a single module between CommonJS and ESM callers in a Node 22+ project.

13. **notion-large-pages.md** → `notion-large-pages`
    > Use when `notion_retrieve_block_children` returns content exceeding token limits and gets saved to a temp file, or when working with Notion pages containing substantial content that might hit that limit.

14. **openrouter-slug-format.md** → `openrouter-slug-format`
    > Use when writing or editing code that passes an `anthropic/...` (or similar) model slug to an OpenRouter SDK call, or when converting a direct Anthropic model ID to an OpenRouter slug.

15. **remote-asset-integrity.md** → `remote-asset-integrity`
    > Use when designing or debugging a pipeline that uploads files to remote storage (R2, S3, CDN) and subsequently triggers processing, rendering, or deployment from that storage.

16. **security-baseline.md** → `security-baseline`
    > Use when starting a new Supabase or Railway project, when writing code that touches authentication, database access, secrets handling, or public API endpoints, or when reviewing for security concerns in an existing project.

17. **supabase-deploy-from-worktree.md** → `supabase-deploy-from-worktree`
    > Use when deploying a Supabase Edge Function while working in a git worktree on a feature branch, or when the edit-to-deploy workflow may put live ahead of main.

18. **supabase-query-patterns.md** → `supabase-query-patterns`
    > Use when writing or debugging Supabase queries via supabase-js, especially insert/update/upsert calls that need error handling, FK-alias embedded joins, or JSONB containment filters.

19. **tracking-and-verification.md** → `tracking-and-verification`
    > Use when starting, working on, or resuming a tracked software project, any task where progress spans multiple steps and needs durable state between sessions, or when closing out a session with verification claims.

20. **youtube-api-gotchas.md** → `youtube-api-gotchas`
    > Use when writing or debugging code that uses the YouTube Data API, especially channel searches, video description text processing, or when passing YouTube metadata downstream to an API that does strict JSON encoding.

### Rules that split into multiple skills

21. **nextjs-server-components.md** → splits into **2 skills:**

    a. `nextjs-server-component-events`
    > Use when authoring a Next.js App Router server component (or debugging a 500-with-opaque-digest failure) that passes `onClick`, `onChange`, or any event handler to a child element, including to `<Link>`.

    b. `nextjs-server-component-verification`
    > Use when verifying a Next.js App Router server component's behavior: testing branching state machines via env-var short-circuits, or smoke-testing Server Actions forms from the command line.

22. **computer-use-tiers.md** → splits into **2 skills:**

    a. `computer-use-tiers`
    > Use when planning or executing browser or desktop automation via the computer-use MCP, when deciding between computer-use, Claude-in-Chrome, and dedicated app MCPs, or when hitting tier-restricted errors (read-only, click-only).

    b. `preview-eval-timeouts`
    > Use when calling `mcp__Claude_Preview__preview_eval` on slow backend work (any fetch over ~20s) or when recovering from a timeout on a write-mutation request (always check for side effects before retrying).

### Rules that stay as reference markdown

These document facts or one-time setup checklists. Auto-triggering adds little value because they're lookup material, not guidance that changes Claude's output.

1. **apple-health-no-rest-api.md** — one-paragraph statement of a fact (no API exists) + the shortcut workaround. Pure reference.

2. **railway-domains.md** — three-step domain setup checklist (CNAME + TXT + port). Looked up per-domain, not triggered by coding patterns.

## Open questions / judgment calls

A handful of these are close. Flagging for final eyeball before the next session writes frontmatter:

1. **openrouter-slug-format** — tiny file (one table + a paragraph), arguably reference. But the silent failure mode is painful enough that auto-triggering when code uses an `anthropic/...` slug is high value. **Current call: skill.** Push back if you want it kept as reference.

2. **nextjs-server-components** split — could also be 3 skills (events / state-machine verification / server-actions curl) instead of 2. Going with 2 because the state-machine env-var trick and the server-actions curl trick both fire in "I'm verifying this works from outside the browser" contexts. **Current call: 2 skills.** Could bump to 3 if Dave wants more granular triggers.

3. **computer-use-tiers** split — alternative is one skill with a broader trigger. Going with 2 because tier awareness fires at tool selection time (early) while `preview_eval` timeout fires during execution (later). **Current call: 2 skills.**

4. **tracking-and-verification** — this is a meta-skill about the whole workflow. The description is broad ("any task where progress spans multiple steps"). Worth keeping as a skill because the workflow discipline is load-bearing, but the trigger may fire more often than needed. **Current call: skill.** Watch for over-triggering when deployed.

5. **security-baseline** — same over-trigger risk. "When writing code that touches auth, database access, secrets, or public API endpoints" covers a lot of real-world coding. **Current call: skill.** Watch in practice.

## Naming notes

Most current filenames translate to skill names directly. Only non-obvious choices:

- `ai-sdk-openai-provider-defaults` → `ai-sdk-provider-defaults` (dropping "openai" since the rule applies to all OpenAI-compatible providers, not just OpenAI itself)
- `nextjs-server-components` split into the two names above (the original filename is too broad for a skill)
- `computer-use-tiers` split as above (original filename becomes one of the two, `preview-eval-timeouts` is new)
