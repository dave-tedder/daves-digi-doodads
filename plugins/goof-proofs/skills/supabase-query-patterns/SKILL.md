# Supabase Query Patterns

Note: the "always destructure `{ error }` on writes" rule lives in the `security-baseline` reference rule, not this skill, since it's general baseline hygiene rather than a pattern-specific gotcha.

## Avoid FK-alias joins in production code

supabase-js supports embedded-resource joins via the PostgREST foreign-key alias syntax:

```ts
.select("id, description, thoughts!action_items_source_thought_id_fkey(metadata)")
```

This is fragile. The constraint name (`action_items_source_thought_id_fkey`) is auto-generated by Postgres at schema-creation time and is invisible in supabase-js types. If the FK is ever renamed, dropped-and-recreated, or the table was created via a migration tool that uses a different naming convention, the query 400s at runtime with no type-check warning. There is no way to verify the constraint name short of querying `pg_constraint` directly:

```sql
SELECT conname FROM pg_constraint
WHERE conrelid = 'public.your_table'::regclass AND contype = 'f';
```

**Prefer a two-query approach** when the code is in production. One query gets the parent rows, a second `.in("id", parentIds)` batches the related rows:

```ts
const { data: items } = await supabase
  .from("action_items")
  .select("id, description, source_thought_id")
  .eq("status", "open");

const sourceIds = Array.from(new Set(items.map((i) => i.source_thought_id)));
const { data: sources } = await supabase
  .from("thoughts")
  .select("id, metadata")
  .in("id", sourceIds);

const sourceById = new Map(sources.map((s) => [s.id, s.metadata]));
```

One extra round-trip, zero schema-name dependency, explicit failure modes, and the related rows come back with their own full selection instead of nested under an alias key.

The FK-alias form is fine for one-off Dashboard/SQL-editor queries where you can see the constraint name and iterate quickly. Don't ship it to an Edge Function.

## `.insert().select("id").single()` when you need the inserted row

Plain `.insert()` returns no data. If post-insert processing depends on the new row's UUID (action item hooks, embedding backfill, downstream triggers), chain `.select("id").single()`:

```ts
const { data: inserted, error } = await supabase
  .from("thoughts")
  .insert(row)
  .select("id")
  .single();

if (error || !inserted) {
  // fail fast, don't try to use inserted.id
}
// inserted.id is the UUID
```

## JSONB containment for upsert dedup

When deduping upserts by a JSONB metadata field (e.g. `metadata.gcal_event_id`), use `.contains("metadata", { key: value })`:

```ts
const { data: existing } = await supabase
  .from("business_events")
  .select("id")
  .contains("metadata", { gcal_event_id: eventId })
  .limit(1);
```

No need for raw SQL or a custom RPC. PostgREST handles the `@>` operator natively.

## Filtering for non-null JSONB array fields

`.not("metadata->dates_mentioned", "is", null)` filters for rows where the JSONB key exists and is non-null, but it returns empty arrays `[]` too. If you need rows where the array has contents, fetch with the null filter and then check `.length > 0` in application code. There is no PostgREST operator for "non-empty array inside JSONB" that supabase-js exposes cleanly — a raw SQL RPC is overkill for most cases.
